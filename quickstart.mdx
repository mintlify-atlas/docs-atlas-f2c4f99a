title: "Quickstart"
description: "Get from zero to running your first browser test."
---

<Note>
  Expect requires **Claude Code**, **Codex CLI**, or **Cursor** to be installed and authenticated. Make sure your `ANTHROPIC_API_KEY` is set (for Claude) or the relevant agent is signed in before running tests.
</Note>

<Steps>
  <Step title="Install Expect">
    The fastest way to install is with `npx`. This installs `expect-cli` globally and sets up the agent skill in one step:

    ```bash
    npx -y expect-cli@latest init
    ```

    Or install the package directly with your preferred package manager:

    <CodeGroup>

    ```bash npm
    npm install -g expect-cli@latest
    ```

    ```bash pnpm
    pnpm add -g expect-cli@latest
    ```

    ```bash yarn
    yarn global add expect-cli@latest
    ```

    ```bash bun
    bun add -g expect-cli@latest
    ```

    </CodeGroup>
  </Step>

  <Step title="Start your dev server">
    Expect runs tests against your live application, so your dev server must be running before you invoke `expect-cli`.

    ```bash
    npm run dev
    ```

    By default, Expect connects to `http://localhost:3000`. If your app runs on a different host or port, set `EXPECT_BASE_URL`:

    ```bash
    export EXPECT_BASE_URL=http://localhost:5173
    ```
  </Step>

  <Step title="Run Expect">
    From your project directory, run:

    ```bash
    expect-cli
    ```

    This opens the interactive TUI where you can review the generated test plan, edit steps, and approve before execution.

    To skip the review and run immediately (useful for agent-driven workflows), pass `-m` with an instruction and `-y`:

    ```bash
    expect-cli -m "test the login flow" -y
    ```

    <Tip>
      Write adversarial instructions rather than confirmation checks. Instead of "check that the login form renders", use "submit the login form empty, with an invalid email, with a wrong password, and with valid credentials — verify error messages and redirect on success."
    </Tip>
  </Step>

  <Step title="Review results">
    After execution completes, Expect prints a pass/fail summary for each test step. For any failures, the output names the exact step and what broke.

    Every run is also recorded with rrweb. The output includes a replay URL you can open in a browser to watch exactly what happened.

    Expect exits `0` on success and `1` on failure, making it suitable for CI pipelines.
  </Step>
</Steps>
